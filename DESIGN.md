# TraceGuard — Design Notes

This document is the engineering narrative behind TraceGuard: why the
architecture is shaped the way it is, the specific decisions each monitor makes,
two real bugs found while building it, and how the validation suite scores
itself. The README covers *what* TraceGuard does and how to run it; this covers
*why* it's built this way.

## 1. One pipeline pattern, used three times

All three monitors are deliberately identical in shape. The point of building
the exec monitor first was to fix a pattern that the file and network monitors
then drop into with no new plumbing:

```
eBPF program (tracepoint)
  -> BPF_MAP_TYPE_RINGBUF                       (per-monitor ring buffer)
    -> ringbuf.Reader                           (one reader goroutine per monitor)
      -> decode raw sample into bpf2go struct   (decodeExec / decodeFile / decodeNet)
        -> build the unified Event              (one struct, fields per type)
          -> shared channel -> rule engine      (single consumer, sole writer to stdout)
```

The kernel side reserves a slot in its ring buffer, fills a fixed-layout
`struct event`, and submits it. The Go side, generated by bpf2go from that same
struct's BTF, reads the raw bytes and decodes them with `binary.Read` into the
generated `execmonEvent` / `filemonEvent` / `netmonEvent`, then flattens that
into one `Event` type (`rules.go`) regardless of which monitor produced it. A
single consumer goroutine owns stdout, so two monitors firing at once can never
interleave a half-written JSON line.

Concretely, adding the network monitor (Step 5) was "a third reader goroutine on
the same channel" — `main.go`'s `run()` adds `netmon` by repeating the same
load → attach tracepoint → open ring buffer → `readLoop` sequence, and the
consumer and rule engine needed nothing new.

## 2. Per-monitor decisions, and why

### execmon — `sched_process_exec`, not execve entry/exit correlation

A natural alternative is to hook `sys_enter_execve` and `sys_exit_execve` and
correlate them to learn whether the exec succeeded and with what. That's two
events and a correlation table. `sched/sched_process_exec` instead fires **once,
only on a successful exec**, after the new program image is in place — exactly
the moment we want, with no correlation bookkeeping and no events for failed
execs to filter out.

### Hand-declared partial structs, not a generated `vmlinux.h`

`execmon.bpf.c` needs a couple of `task_struct` fields (`real_parent`, `tgid`,
`comm`); the file and net monitors need the generic
`trace_event_raw_sys_enter` context. The usual approach is `bpftool btf dump ...
> vmlinux.h` — a multi-megabyte header describing the entire kernel. We instead
hand-declare *only* the fields we touch, marked
`__attribute__((preserve_access_index))` so CO-RE still relocates their real
offsets against the running kernel's BTF at load time. This keeps the build
self-contained with the toolchain already in use (clang + libbpf headers), with
no `bpftool`-generated artifact to regenerate per kernel.

### netmon — `sys_enter_connect`, not a kprobe on `tcp_v4_connect`

Hooking the syscall entry lets us read the `sockaddr` straight from the
userspace argument pointer (`args[1]`) with `bpf_probe_read_user`. The
alternative — a kprobe on `tcp_v4_connect` — would mean reading destination
address and port out of kernel `struct sock` internals, which is exactly the
kind of unstable, version-sensitive layout CO-RE exists to paper over. Reading
the userspace `sockaddr_in` at the syscall boundary **avoids needing CO-RE
against kernel socket internals at all**; we read the first 8 bytes
(family/port/addr) of a trimmed `sockaddr_in` and bail out unless `sin_family ==
AF_INET`.

### filemon — an in-kernel noise filter, and the `#pragma unroll` trap

`openat()` is extremely high-volume; forwarding every open to userspace would
bury the signal and waste ring-buffer bandwidth. So `filemon.bpf.c` applies a
**coarse pre-filter in the kernel** and forwards only opens that are plausibly
interesting: any *write* open (any path), reads under `/etc/`, or reads of paths
carrying a credential marker (`.ssh/`, `.aws/`, `.pem`, `id_rsa`, `id_ed25519`).
Everything else is `bpf_ringbuf_discard`ed before it ever crosses into Go. The
kernel stays a coarse filter; the precise decision (which exact path matters) is
left to the rule engine.

The credential-marker scan (`has_sensitive_marker`) is a bounded substring
search over the 256-byte path. Its design is shaped by a BPF-specific
constraint:

> An earlier version used `#pragma unroll` over the scan loops. The verifier
> rejected it. Fully unrolling the outer scan (up to 248 positions) with the
> nested per-marker character comparisons expands into **thousands of inlined
> comparisons**, and the intermediate state blows past BPF's **512-byte stack
> frame limit** — even though the program declares no large arrays. Removing the
> `#pragma unroll` and leaving a plain *bounded* loop fixes it: since kernel
> 5.3 the verifier handles bounded loops natively, walking the loop body once
> and proving termination, without needing it unrolled. Every index is also
> masked with `& 0xff` so the verifier can prove each path access is in-bounds.

## 3. Two bugs worth writing down

### The `/proc`-based parent-name race (fixed in commit 73af29f)

`evalShellSpawn` needs the *parent's* command name to decide whether a shell was
spawned by an allowed parent. The first implementation resolved it in userspace,
after the event arrived: read `/proc/<ppid>/comm`. This has a race. Fast exec
chains — exactly the reverse-shell / dropper patterns we most want to catch —
can have a parent that **has already exited** by the time the event reaches Go
and we go looking in `/proc`. The lookup then resolves to `(unknown)`.

What made this nasty wasn't the missing name itself — it was that `(unknown)`
was **indistinguishable from genuine background noise** that produced the same
symptom, so the failure looked like ambient flakiness rather than a race.

The fix moves parent-name capture **into the kernel**, read at the exact instant
of the exec event via `BPF_CORE_READ_STR_INTO(&e->parent_comm, task,
real_parent, comm)`. At that moment the parent is, by definition, still alive
(it's the parent of the process that just `exec`'d). This eliminates the race
outright rather than suppressing the symptom — there is no later lookup to lose
to a race.

### The credential-path kernel-filter gap (found by the validation suite, fixed in fc29078)

The validation suite added a `credential-file-evasion` case: a read of
`/root/.aws/credentials`. The rule engine was correct — `sensitive_files`'
`path_substrings` already listed `.aws/`. But the alert never fired.

The cause was a layering gap. `filemon.bpf.c`'s kernel-side noise filter
originally forwarded only **writes or `/etc/` reads**. A *read* of an `.aws/`
credential file outside `/etc/` matched neither, so the kernel discarded it
**before the rule engine ever saw it.** The rule's config was right but
unreachable for that case.

The fix extends the kernel filter itself (`has_sensitive_marker`) to also
forward reads matching known credential markers. Crucially this keeps the
division of labor intact: the kernel remains a *coarse* pre-filter that only has
to avoid dropping anything the rule engine might match, while the rule engine
still makes the precise final decision. We widened the kernel's net, we didn't
move policy into the kernel.

## 4. Rule engine: four purpose-built sections, not a DSL

The rule engine (`rules.go`) is four hand-written evaluators —
`evalShellSpawn`, `evalSensitiveFiles`, `evalReadonlyWrites`,
`evalNetworkAnomaly` — each reading its own typed YAML section. The tempting
alternative is a generic match-operator DSL (fields, operators, boolean
combinators) that rules are expressed in.

We chose the purpose-built sections deliberately: a generic DSL is a parser, an
evaluator, an operator set, and a class of "the rule file is syntactically valid
but means something subtly wrong" mistakes — all to express four well-understood
categories. Four small typed structs and four `eval*` functions are **less code
and less to get wrong**, while still covering every category v1 needs. Each
category exposes exactly the knobs that make sense for it (allowlists of parents,
path substrings, protected prefixes, allowed ports) rather than forcing all of
them through one generic shape. If the rule set ever outgrows this, a DSL is a
v2 conversation — not a v1 default.

## 5. Validation methodology — and what it does and doesn't prove

The harness (`validate/`) runs a fixed suite of attack and benign commands while
TraceGuard is running with `-log-file`, then scores by **time-window matching**:

- Each test records the wall-clock window `[Started, Ended]` around its command
  (the command, plus a 1.5s buffer for the event to propagate through the ring
  buffer → Go → rule eval → log-write pipeline).
- After the run, it loads every `"type":"alert"` line from the log and attributes
  an alert to a test if the alert's timestamp falls inside that test's window
  (`scoreTestResult`).
- An **attack** passes if its expected rule appears in its window; a **benign**
  case passes only if its window is *empty*. Detection rate = passed attacks /
  attacks; false-positive rate = benign cases that fired / benign cases.

The suite is carefully built to avoid measuring its own artifacts: shell-spawn
attacks run as direct `argv` (not wrapped in a host `bash -c`, which would itself
trip `unexpected-shell-spawn` from the validator's own parent), benign container
cases use non-shell entrypoints, and the `zsh`-enabled victim container is set up
*before* the timed loop so the `apt-get install`'s own shells land outside every
window.

What this approach **proves**: against this specific suite, every attack
technique (including the two evasion variants) is detected, and no benign case
fires — on one machine, one run.

What it **does not prove**:

- It's a **single sequential run**, not repeated trials — there's no statistical
  confidence interval, just one pass.
- Time-window attribution can in principle mis-attribute an alert from unrelated
  background activity that happens to land in a window; the suite mitigates this
  by construction but doesn't eliminate it.
- The accuracy tests run with **no concurrent or bursty load**. CPU overhead
  under burst is measured *separately* by `perf_test.sh` /
  `perf_steadystate.sh`; the accuracy run and the perf run are different
  experiments and shouldn't be read as one.

## 6. Non-goals for v1

The v1 scope boundaries — detect-only (no blocking/remediation), `openat()` over
legacy `open()`, IPv4-only networking, comm-name (not per-cgroup) container
attribution, no cross-monitor correlation, and no `chmod`/`ptrace`/module-load
coverage — are enumerated with their rationale in the README's **Known
limitations** and **Scope** sections. They are deliberate boundaries for v1, not
oversights; see there rather than duplicating them here.
